# Quality Convention

Cross-cutting quality rules for all skills that write files or produce output.

Adapted from [gbrain's quality convention](https://github.com/garrytan/gbrain).

---

## Output Precision

Every generated or modified file must be minimal and correct:

- Make the smallest edit that achieves the goal — never rewrite a file to change one line
- Preserve surrounding formatting (table alignment, list indentation, heading levels)
- Never add content the user did not ask for (extra sections, comments, TODO markers)

## Notability Gate

Before creating a new file or skill, check whether it is worth creating:

- **Skills**: Will this save ≥5 minutes of real work? Does an existing skill already cover it?
- **Docs**: Will this be referenced again? Does it stay current without manual effort?
- **Memory**: Will this change how a future session approaches the project?

When in doubt, don't create. A question in the conversation beats a stale file.

## Consistency Rules

- Follow the naming and formatting conventions already established in the file being edited
- Match heading levels, bullet style, and code block language tags to surrounding content
- If the file uses em dashes, use em dashes. If it uses hyphens, use hyphens. Do not mix.

## No AI Attribution in Output

- No "Generated by Claude", "AI-generated", "Co-Authored-By: Claude" in any file
- No emoji in output unless the file already uses emoji
- No AI-tell vocabulary: avoid *delve*, *leverage*, *utilize*, *seamless*, *robust*, *holistic*

## Verification Step

After writing or editing any file, verify:

```bash
# Confirm the file exists and is non-empty
wc -l <file>

# Confirm no merge conflict markers were introduced
grep -c "<<<<<<< " <file> || true

# For shell scripts: confirm syntax
bash -n <file>

# For Python: confirm syntax
python3 -m py_compile <file>
```

## Token Discipline

Every skill must document its expected token range. If a skill's actual usage exceeds its documented range by more than 50%, update the Token Optimization section.
